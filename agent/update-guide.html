🚀 Agent update & rollout guide

How the supervisor updates the worker safely, and how to roll a new build to the fleet. Click Copy on any command.

How it works

agentsvc.exe runs Agent.exe · push update → worker stages Agent.new.exe + exits → supervisor backs up & swaps → probation (worker.hb < ~90s) → healthy ✅ or auto-rollback to Agent.prev.exe

The boot task runs the supervisor (agentsvc.exe), which child-manages the worker (Agent.exe). The worker is never the boot exe, so it can be swapped freely — no file-lock or self-kill problem. During the swap the supervisor keeps the PC online (keepalive ping), so the connection isn't lost. A bad build that doesn't check in is rolled back automatically.

Prereqs: the VPS has pulled the latest (?action=ping + ?action=update), and you know your API key (from the VPS config.php). Build the new exes with build.bat and bump AGENT_VERSION (date-based, e.g. 2026.06.22).

① One-time migration — switch a PC to the supervisor modelfirst time only

Existing PCs run the old Agent.exe as the boot task. Move agentsvc.exe + the new Agent.exe next to it (Files page upload), then run this in the web Terminal (PowerShell, SYSTEM). It re-points the boot task to the supervisor via a detached task so killing the agent doesn't kill the updater:

$task='rmdownloaderAgent' $dir = Split-Path -Parent (Get-ScheduledTask -TaskName $task).Actions.Execute $bat='C:\Windows\Temp\rmmig.bat' @" @echo off schtasks /end /tn "$task" ping -n 3 127.0.0.1 >nul start "" "$dir\agentsvc.exe" schtasks /delete /tn rmmig /f "@ | Set-Content -Encoding ascii $bat schtasks /create /tn rmmig /tr "cmd /c $bat" /sc once /st 00:00 /ru SYSTEM /rl HIGHEST /f schtasks /run /tn rmmigCopy
→ the old worker stops, agentsvc.exe starts, takes over the boot task (re-creates it pointing at itself when elevated), and launches the worker. The PC reappears online. Do this on a canary PC first.

② Roll out a new worker build — canary, then fleetevery update after

From this machine (the build box, which has the new Agent.exe), run the rollout script. It pushes to one canary, waits for it to reconnect on the new version, and only then does the rest. A PC that doesn't come back on the new version is reported as failed (its supervisor rolls it back).

powershell -ExecutionPolicy Bypass -File C:\Users\Administrator\Desktop\rmdownloader\agent\rollout-update.ps1 -Server https://dos.argonar.co -ApiKey YOUR_API_KEY -ExePath C:\Users\Administrator\Desktop\rmdownloader\agent\Agent.exe -Version 2026.06.22Copy
→ === CANARY === … OK, then === FLEET === per-PC OK/FAILED, then a summary. Pin the canary with -CanaryId <id>; widen the wait with -TimeoutSec 180.

③ Update a single PC manually (optional)

Without the script — base64 the exe and POST it to the update action for one agent:

$b64 = [Convert]::ToBase64String([IO.File]::ReadAllBytes('C:\Users\Administrator\Desktop\rmdownloader\agent\Agent.exe')) Invoke-RestMethod -Method Post -Uri "https://dos.argonar.co/api.php?action=update&agent=AGENT_ID&key=YOUR_API_KEY" -Body @{ exe=$b64; version='2026.06.22' }Copy
→ {ok:true, staged:true, ...}; the supervisor swaps + health-checks within ~90s.

④ Verify

List agents and their reported versions (the picker tooltip shows this too):

Invoke-RestMethod "https://dos.argonar.co/api.php?action=agents&key=YOUR_API_KEY" | Select-Object -ExpandProperty agents | Format-Table name,online,versionCopy
→ updated PCs show version = 2026.06.22 and online = True.
Safety: always canary first. A bad worker rolls itself back per-PC, but a build that connects yet misbehaves won't trigger rollback — so confirm the canary actually works (open its files, run a command) before the fleet.
Updating the supervisor itself (rare) isn't done via the rollout — use the migration recipe in step ① (it overwrites agentsvc.exe the same way). The supervisor is intentionally tiny so this is seldom needed.
