🌐 TinyDNS — self-install / boot-task checker

Run these on the DNS machine to confirm dnl.exe registered its boot task and is serving. Click Copy on any command, paste into an Administrator PowerShell.
Open PowerShell as admin: press Win, type powershell, right-click → Run as administrator. The boot task is created as SYSTEM, so checks that read it want elevation.

1 · Is the boot task installed?the main check

Lists the TinyDNS task with all details.

schtasks /query /tn TinyDNS /v /fo listCopy
Installed → you get a block with TaskName: \TinyDNS, Status: Running (or Ready), Run As User: SYSTEM, Logon Mode: ....   Not installed → "ERROR: The system cannot find the file specified."

2 · Does it start at every boot? (boot trigger + run level)

Shows the trigger, principal and whether it's hidden — proves "every boot, as SYSTEM, hidden".

Get-ScheduledTask -TaskName TinyDNS | Select-Object -ExpandProperty Triggers; Get-ScheduledTask -TaskName TinyDNS | ForEach-Object { $_.Principal; $_.Settings.Hidden }Copy
Good → a trigger of type MSFT_TaskBootTrigger (At startup), UserId: S-1-5-18 / RunLevel: Highest, and Hidden: True.

3 · Which exe does the task run?

Confirms the task points at your dnl.exe (and tells you the folder it reads records/blocklist from).

(Get-ScheduledTask -TaskName TinyDNS).Actions | Select-Object Execute, WorkingDirectoryCopy
→ Execute is the full path to dnl.exe.

4 · Is the process actually running?

tasklist | findstr /i dnl.exeCopy
Running → one dnl.exe line with a PID. (It's windowless — you'll never see a console.)

5 · Is it listening on DNS port 53?

netstat -ano -p udp | findstr :53Copy
→ a line UDP 0.0.0.0:53 ... <PID> whose PID matches step 4.

6 · Does it actually resolve?

Asks the server itself. A local record returns instantly; a normal site is forwarded.

nslookup example.local 127.0.0.1Copy
nslookup google.com 127.0.0.1Copy
→ example.local gives the IP from records.txt; google.com returns a real address (forwarded).

7 · Install it (if step 1 said "not installed")

The exe self-registers the boot task the first time it runs elevated. Just run it once as admin:

Start-Process "C:\path\to\dnl.exe" -Verb RunAsCopy

…then re-run step 1 — the TinyDNS task should now exist. (Double-clicking the exe and accepting the UAC prompt does the same thing.)

8 · Remove the boot task

Either ask the exe to remove its own task, or delete it directly:

"C:\path\to\dnl.exe" --uninstallCopy
schtasks /end /tn TinyDNS; schtasks /delete /tn TinyDNS /fCopy
Tip: to swap in a new build, /end the task first (don't just kill the process — RestartOnFailure may relaunch the old exe), overwrite dnl.exe, then schtasks /run /tn TinyDNS.
